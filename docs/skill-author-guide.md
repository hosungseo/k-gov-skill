# 스킬 작성 가이드

좋은 스킬은 “무엇을 할 수 있다”보다 “어떤 행정업무 병목을 줄인다”를 먼저 말합니다.

## 1. 이름

짧고 기능이 드러나게 씁니다.

좋음:
- `public-data-portal`
- `official-gazette-search`
- `org-quota-review`

나쁨:
- `super-ai-helper`
- `government-automation`

## 2. 반드시 적을 것

각 스킬 README에는 아래 항목이 있어야 합니다.

- 목적
- 사용자 신청/로그인
- 필요한 환경변수
- 신청처/출처
- 사용 예시 프롬프트
- 예상 산출물
- 키 없을 때 fallback
- 주의

## 3. 인증/키를 숨기지 않기

공공데이터포털, 국회, KOSIS, DART처럼 사용자가 직접 신청해야 하는 API는 “필요” 또는 “선택”으로 표시합니다.

나쁜 문장:
> 이 스킬은 공공데이터포털 API를 바로 호출합니다.

좋은 문장:
> 공식 API 호출에는 사용자가 직접 발급받은 `DATA_GO_KR_SERVICE_KEY`가 필요합니다. 키가 없으면 신청 방법과 샘플 출력만 제공합니다.

## 4. 산출물은 업무 패킷으로

가능하면 결과를 아래 구조로 냅니다.

```text
요청 요약
확인한 공식 출처
주요 파라미터/근거
쟁점
불확실한 부분
사람 검토 필요 지점
붙여넣을 수 있는 초안
```

## 5. 안전 경계

- 개인정보와 민감정보를 최소 처리합니다.
- 법령 해석, 처분 판단, 외부 제출은 사람 검토를 요구합니다.
- 원문 링크, 기관명, 기준일을 남깁니다.
